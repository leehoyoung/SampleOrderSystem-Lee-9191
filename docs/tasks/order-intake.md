## 기능: 시료 주문 접수/예약 (PRD 7.3)

### 선행 조건

- `Sample/` 도메인(등록/조회, `Sample::id`/`Sample::stock`)은 이미 스캐폴딩되어 있고 이번
  스펙에서는 "조회 전용"으로만 참고한다. 시료 등록/조회/검색(7.2) 자체는 이번 스펙 범위가
  아니다.
- `Order/OrderModel::createOrder(sampleId, customerName, quantity)`와 `getAll()`은 이미
  포팅되어 RESERVED 상태로 주문을 생성한다(`SampleOrderSystemTests/OrderModelTest.cpp`에
  characterization 테스트 존재). 이 로직 자체는 재작성하지 않되, 아래 태스크 1에서
  PRD 6.1(재고는 승인/생산완료 시점에만 갱신)과 실제로 부합하는지 TDD로 명시적으로
  못박는다.
- `Order/OrderController::execute()`는 완전 스텁("준비 중입니다" 출력)이다. 이번 스펙에서
  실제 메뉴 흐름(주문 접수 입력 → 생성, 목록 표시, 종료)으로 채운다.
- 이번 스펙 범위는 PRD 7.3만이다. **승인/거절(7.4), 생산 라인(7.6), 재고 갱신(6.1의 승인
  이후 갱신 규칙), 모니터링(7.5), 출고(7.7)는 제외**하고 이후 별도 스펙으로 진행한다.
  이번 스펙에서 확인하는 "목록 표시"는 7.4가 정의하는 "RESERVED 필터 목록"이 아니라
  `OrderModel::getAll()`이 반환하는 전체 주문 목록이다(이 스펙 시점에는 승인 기능이
  없으므로 모든 주문이 사실상 RESERVED와 동치이다).
- 재사용: `Order/OrderRepository`, `Sample/SampleRepository`, `Sample/SampleModel`
  (DataPersistence PoC의 Repository 패턴을 그대로 따름 — 새 저장소 포맷을 만들지 않는다).
  시료 존재 확인을 위해 `SampleModel`/`SampleRepository`에 ID 조회 메서드(예:
  `findById`)가 없으면 이번 스펙에서 추가가 필요하다(구현 방식은 implementer 재량).
- 순서: (1) 기존 RESERVED 생성 로직의 재고 미변경 재검증 → (2) 시료 존재 검증 로직
  (Model 계층) → (3) 신규 주문 생성 성공 경로(검증 포함) → (4) Controller/View 메뉴 흐름
  (입력 → 생성 → 목록 표시 → 종료/재입력 루프) 순으로 진행한다. Controller/View 계층은
  "확인 필요 사항"의 테스트 전략 결정 후 착수한다.

### TDD 태스크 목록 (순서대로)

1. **[재검증] `createOrder` 호출은 대상 시료의 재고 필드를 변경하지 않는다.**
   — 입력: 시료 `S-001`(재고 30)이 등록된 상태에서
   `OrderModel::createOrder("S-001", "고객사", 10)` 호출.
   기대 결과: 반환된 주문의 상태는 `RESERVED`이고, 이후 `SampleRepository`/`SampleModel`로
   조회한 `S-001`의 `stock`은 여전히 30 (변화 없음).
   근거: PRD 6.1 "재고 필드는 승인 시점 또는 생산완료 시점에만 갱신된다". 기존
   characterization 테스트(`OrderModelTest.CreateOrderAlwaysStartsAsReservedWithNextOrderNoFormat`)는
   상태/주문번호 형식만 검증하고 재고 불변은 검증하지 않으므로 새 RED 테스트로 보강한다.

2. **[경계] 등록되지 않은 시료 ID로는 주문이 생성되지 않는다.**
   — 입력: 등록된 시료가 `S-001`뿐인 상태에서 `sampleId="S-999"`, `customerName="ACME"`,
   `quantity=10`으로 주문 생성 시도.
   기대 결과: 주문이 생성되지 않는다(주문 생성 전후 `OrderModel::getAll().size()` 불변),
   호출자에게 실패를 알 수 있는 신호(예외/오류 반환값 — 형식은 implementer 결정)가 전달된다.
   근거: PRD 5.1 "시스템에 등록된 시료만 주문 가능하다".

3. **[경계] 등록된 시료가 하나도 없는 상태에서 주문 생성을 시도해도 거부된다.**
   — 입력: 시료 미등록 상태(`SampleRepository` 빈 목록)에서 `sampleId="S-001"`로 주문
   생성 시도.
   기대 결과: 태스크 2와 동일하게 주문이 생성되지 않는다. (시료 0건이라는 극단값을
   별도로 고정하기 위한 태스크.)

3-1. **[경계, 확정] `quantity <= 0`인 주문은 생성되지 않는다.**
   — 입력: 시료 `S-001`(등록 상태)에 대해 `quantity=0`, `quantity=-5`로 각각 주문 생성
   시도.
   기대 결과: 두 경우 모두 주문이 생성되지 않는다(`getAll().size()` 불변), 호출자에게
   실패 신호 전달(형식은 implementer 결정). 재고 상한(주문 수량이 재고를 초과하는
   경우)은 PRD 6.1상 재고 부족 자체가 PRODUCING으로 정상 처리되는 케이스이므로 이번
   태스크에서 검증하지 않는다.
   근거: 사용자 확정 — "quantity <= 0은 접수 단계에서부터 거부".

4. **등록된 시료 ID로 주문을 생성하면 RESERVED 상태 주문이 만들어진다.**
   — 입력: 시료 `S-001`(재고 무관, 임의 값) 등록 상태에서 `sampleId="S-001"`,
   `customerName="ACME 연구소"`, `quantity=15`로 주문 생성.
   기대 결과: 주문 1건 생성, `status == RESERVED`, `orderNo`가 `"ORD-"` 접두 형식,
   `sampleId`/`customerName`/`quantity`가 입력값과 일치. (태스크 2·3에서 도입한 검증
   로직이 정상 케이스를 막지 않는지 확인하는 회귀 테스트.)

5. **주문 관리 메뉴 진입 시 하위 메뉴(주문 접수/목록 조회/뒤로가기)가 표시된다.**
   — 입력: 메인 메뉴에서 "주문 관리" 진입.
   기대 결과: `OrderView`가 최소 "주문 접수", "접수된 주문 목록", "뒤로가기"에 대응하는
   선택지를 표시한다(문구는 자유, 선택지 개수/의미만 고정).
   테스트 전략은 "확인 필요 사항" 항목 결정 후 확정한다.

6. **주문 접수 메뉴에서 시료ID/고객명/수량을 입력하면 주문이 생성되고 발급된 주문번호가
   화면에 표시된다.**
   — 입력: 등록된 시료 `S-001` 존재, 사용자 입력 순서 `sampleId="S-001"`,
   `customerName="ACME"`, `quantity=10`.
   기대 결과: `OrderModel`에 RESERVED 주문 1건 생성, 화면에 발급된 주문번호가 포함된
   완료 메시지 출력.

7. **[경계] 주문 접수 메뉴에서 존재하지 않는 시료 ID를 입력하면 주문이 생성되지 않고
   실패 메시지가 표시된다.**
   — 입력: 등록된 시료가 `S-001`뿐인 상태에서 `sampleId="S-999"` 입력.
   기대 결과: 주문이 생성되지 않음(태스크 2 로직 재사용), 화면에 실패를 알리는 메시지
   출력, 메뉴로 복귀(프로그램 종료되지 않음).

8. **접수된 주문 목록 조회 메뉴는 현재까지 생성된 모든 주문을 표시한다.**
   — 입력: 주문 3건(서로 다른 시료ID/고객명/수량) 생성된 상태에서 "접수된 주문 목록"
   선택.
   기대 결과: 3건 모두 화면에 표시(각 항목에 주문번호/시료ID/고객명/수량/상태 포함).

9. **[경계] 주문이 하나도 없는 상태에서 목록 조회 시 빈 목록임을 알 수 있는 결과를
   보여준다.**
   — 입력: `OrderModel::getAll()`이 빈 벡터인 상태에서 목록 조회 선택.
   기대 결과: 오류 없이 "주문 없음" 등 빈 상태를 알리는 표시(0건 목록 출력도 허용,
   크래시/예외 없음만 고정).

10. **주문 관리 메뉴에서 "뒤로가기(0)" 선택 시 메인 메뉴로 복귀한다.**
    — 입력: 메뉴 진입 후 "0" 선택.
    기대 결과: `OrderController::execute()`가 반환되어 메인 메뉴 루프로 돌아간다
    (기존 `SampleController::execute()`의 0번 처리 패턴과 동일).

11. **[경계] 정의되지 않은 메뉴 번호/문자를 입력하면 오류 메시지를 표시하고 같은 메뉴를
    재표시한다.**
    — 입력: 메뉴에서 "9" 또는 "abc" 입력.
    기대 결과: "잘못된 선택입니다" 류 메시지 출력 후 프로그램이 종료되지 않고 같은
    메뉴를 다시 보여준다 (`SampleController`의 기존 처리와 동일 패턴).

### 확인 필요 사항 (결정 완료)

- **수량(`quantity`) 유효성 검증 규칙 — 확정.** `quantity <= 0`은 접수 단계에서부터
  거부한다 (태스크 3-1). 재고 초과분에 대한 상한 검증은 두지 않는다 — PRD 6.1상 재고
  부족은 PRODUCING으로 정상 처리되는 케이스이기 때문이다.

- **Controller/View 계층 테스트 전략 — 확정 (재변경).** 아직 태스크 5~11이 미착수
  상태에서 사용자가 방침을 다시 확정했다: `IView`를 인터페이스로 추출하고 테스트용
  Fake/Mock을 주입하는 방식으로 진행한다 (표준입출력 리디렉션 방식은 채택하지 않음).
  - `Order/OrderView`(및 필요 시 참고용으로 `Sample/SampleView`의 기존 구조도 확인)를
    `IView`(또는 `IOrderView`) 인터페이스로 추출하는 작업이 선행되어야 하며, 이 추출
    자체도 하나의 TDD 행동 단위로 다룬다(미리 리팩터링해두고 예외 취급하지 않는다).
  - 태스크 5~11의 GTest는 `OrderController`에 실제 콘솔 I/O 대신 테스트 더블(Fake
    View, 필요시 gmock 기반 Mock)을 주입해 호출/표시 내용을 검증하는 방식으로
    작성한다.
  - `SampleController`/`SampleView`는 이번 스펙 범위가 아니므로 건드리지 않는다
    (참고만 하고 리팩터링하지 않음).

- **주문 접수 메뉴 실패 시 메시지 문구/반환 방식**(예외 vs 반환값 vs 화면 메시지만)은
  이번 문서에서 의도적으로 정하지 않았다 — "무엇을 검증해야 하는가"(주문 미생성 +
  실패를 알 수 있는 신호)만 고정하고, 구체적 방식은 implementer 재량으로 남긴다.
