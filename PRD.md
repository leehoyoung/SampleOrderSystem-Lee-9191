# PRD: 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

## 1. 배경

가상의 반도체 회사 "S-Semi"는 연구소, 팹리스 업체, 대학 연구실 등에 반도체 시료(Sample)를
생산·납품한다. 주문이 들어오면 웨이퍼 공정 설비를 통해 시료를 제작하고, 검수 후 고객에게
출고한다.

기존에는 엑셀·메모장으로 주문을 관리하여 처리 누락, 재고/공정 현황 파악 곤란 등의 문제가
있었다. 이를 해결하기 위해 콘솔 기반의 "반도체 시료 생산주문관리 시스템"을 개발한다.

## 2. 목표

- 시료 등록/조회/검색, 주문 접수/승인/거절, 생산 라인 관리, 재고 모니터링, 출고 처리를
  하나의 콘솔 애플리케이션에서 수행할 수 있도록 한다.
- Agentic Engineering(서브에이전트 + 스킬 기반 개발 프로세스)을 적용하여 문서화, TDD,
  Clean Code, 커밋 이력 관리 등 고품질 개발 프로세스를 실천한다.

## 3. 역할 및 흐름

| 역할 | 설명 |
|------|------|
| 고객 | 필요한 시료를 이메일 등으로 요청 (시스템 밖에서 발생하는 행위) |
| 주문 담당자 | 요청에 맞게 주문서(주문) 작성/접수 |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인/거절, 생산 관리 |

```
고객 --(시료 요청)--> 주문 담당자 --(주문서 전달)--> 생산 담당자
                          <--------(승인/거절)---------
```

## 4. 시스템 개요

- 콘솔 기반 애플리케이션. 담당자가 메뉴에서 번호를 선택하여 명령을 입력한다.
- 생산 라인은 공장에서 시료 하나를 생산하는 설비 흐름이며, **단일 생산 라인**만 존재한다.
  생산 라인은 한 번에 하나의 시료만 생산할 수 있다.
- 생산 라인은 주문이 들어온(승인된) 시료에 대해서만 가동된다.
- 개발 언어: C++ (Visual Studio, GTest). 콘솔 MVC 구조 및 데이터 영속성 방식은 별도로
  진행 중인 PoC(ConsoleMVC, DataPersistence)의 산출물을 참고하여 반영한다. 자세한 규칙은
  `CLAUDE.md`를 따른다.

## 5. 도메인 모델

### 5.1 시료 (Sample)

| 속성 | 설명 |
|------|------|
| 시료 ID | 고유 식별자 (예: S-001) |
| 이름 | 시료명 |
| 평균 생산시간 | 1개 생산에 걸리는 평균 시간 (min/ea) |
| 수율 | 정상 생산 비율 = 정상 생산 시료 수 / 총 생산 시료 수 (예: 0.9) |
| 현재 재고 | 출고 가능한 재고 수량 |

- 시스템에 등록된 시료만 주문 가능하다.

### 5.2 주문 (Order)

| 속성 | 설명 |
|------|------|
| 주문번호 | 고유 식별자 (예: ORD-YYYYMMDD-NNNN) |
| 시료 ID | 주문 대상 시료 |
| 고객명 | 주문한 고객 |
| 주문 수량 | 요청 수량 |
| 상태 | 아래 주문 상태 흐름 참조 |

## 6. 주문 상태 흐름

| 상태 | 의미 |
|------|------|
| RESERVED | 주문 접수 (초기 상태) |
| REJECTED | 주문 거절 — 정상 흐름 밖의 상태이며 모니터링에서 제외 |
| PRODUCING | 주문 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 + 출고 대기 중 |
| RELEASED | 출고 완료 |

상태 전이:

```
RESERVED --(거절)--> REJECTED
RESERVED --(승인, 재고 충분)--> CONFIRMED --(출고)--> RELEASED
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED --(출고)--> RELEASED
```

### 6.1 재고 갱신 및 선점 규칙

- **재고(현재 재고) 필드는 다음 두 시점에만 갱신된다.** 그 외(생산 진행 중)에는 값이
  바뀌지 않는다.
  - 승인 시 재고가 충분한 경우: 즉시 `재고 -= 주문 수량`
  - 재고 부족으로 생산 중이던 주문이 완료되어 `CONFIRMED`로 전환되는 시점:
    `재고 = 재고 + 실 생산량 − 주문 수량`
- **신규 주문의 재고 충분/부족 판정은 항상 판정 시점의 재고 필드 값을 그대로 사용한다.**
  별도의 "선점량 차감" 계산은 하지 않는다 — 재고 부족 판정이 나는 순간 그 시점의 재고
  전량이 자연히 그 주문에 귀속되므로(재고 < 주문 수량이라 남는 게 없음), 그 주문이
  확정되기 전까지는 다른 신규 주문이 같은 재고를 사용할 수 없다. 이 귀속 상태는 별도
  처리 없이, 재고 부족 승인 시 재고가 변하지 않는다는 사실 자체로 자연스럽게 유지되고,
  해당 주문이 확정되어 재고 필드가 갱신되는 순간 해소된다.
- 위 규칙에 따라, 이미 `PRODUCING`인 주문이 존재해도 **그 주문이 재고 부족 승인 시점에
  기존 재고를 전혀 쓰지 못했다면**(즉 승인 당시 가용 재고가 이미 0이었다면) 그 주문은
  향후 재고에 아무 영향도 주지 않는다. 예:
  1. 재고 30, 1번 주문(수량 50) 승인 → 부족분 20, 실 생산량 40 → `PRODUCING`.
     이 시점 재고 30 전체가 1번 주문에 귀속된다 (재고 필드는 그대로 30).
  2. 2번 주문(수량 10) 도착 → 가용 재고 0(이미 1번 주문에 귀속) → `PRODUCING`
     (부족분 10 전량 생산, 재고 필드는 건드리지 않음).
  3. 1번 주문 생산 완료 → `CONFIRMED`. 재고 갱신: `30 + 40 − 50 = 20`.
  4. 3번 주문(수량 15) 도착 → 갱신된 재고 20을 그대로 기준으로 판정 → 충분 → 즉시
     `CONFIRMED`. 2번 주문이 여전히 `PRODUCING`(생산 대기/진행 중)이어도, 2번 주문은
     애초에 가용 재고 0일 때 승인되어 재고를 귀속받은 적이 없으므로 이 갱신된 20에
     아무 영향을 주지 않는다.

## 7. 기능 명세

### 7.1 메인 메뉴

전체 시료에 대한 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산 라인 대기 등)를
표시하고, 아래 기능으로 진입하는 선택 화면을 제공한다.

### 7.2 시료 관리

| 기능 | 설명 |
|------|------|
| 시료 등록 | 시료 ID, 이름, 평균 생산시간, 수율을 입력하여 신규 시료 추가 |
| 시료 조회 | 등록된 모든 시료 목록과 현재 재고 수량 표시 |
| 시료 검색 | 이름 등 속성으로 특정 시료 검색 |

### 7.3 시료 주문 (예약)

고객 요청에 따라 주문 담당자가 주문을 생성한다.

- 입력 값: 시료 ID, 고객명, 주문 수량
- 생성 직후 주문 상태는 `RESERVED`

### 7.4 주문 승인/거절

- **접수된 주문 목록**: `RESERVED` 상태 주문 목록 표시
- **주문 승인**: 재고 상황에 따라 자동 분기
  - 재고 충분 → 즉시 `CONFIRMED` 전환
  - 재고 부족 → 생산 라인 큐에 자동 등록, `PRODUCING` 전환
- **주문 거절**: 즉시 `REJECTED` 전환

### 7.5 모니터링

- **주문량 확인**: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASED`) 주문 수 확인.
  `REJECTED`는 유효하지 않은 주문이므로 집계에서 제외한다.
- **재고량 확인**: 시료별 현재 재고 필드 값을 그대로 표시하고, 다음 기준으로 상태를
  표기한다 (주문 수량과 비교하지 않는다):
  - **여유**: 해당 시료의 주문이 생산 라인 큐(`PRODUCING`)에 없고, 재고 > 0
  - **부족**: 해당 시료의 주문이 생산 라인 큐(`PRODUCING`)에 하나 이상 있음
  - **고갈**: 해당 시료의 주문이 생산 라인 큐에 없고, 재고 == 0
  - 재고 수치 자체는 6.1장의 갱신 규칙에 따른 값 그대로다. `PRODUCING` 주문이 그
    재고를 귀속받은 상태여도, 필드가 실제로 갱신되기 전까지는 화면에 이전 값 그대로
    표시한다 (임의로 0으로 보이게 하거나 선점량을 별도로 빼지 않는다).

### 7.6 생산 라인

- 승인 시 재고 부족분이 생산 큐에 등록되며, **선입선출(FIFO)** 로 처리된다.
- 생산 라인은 한 번에 하나의 주문(시료)만 생산한다.
- 실 생산량과 총 생산 시간 계산:
  - 부족분 = 주문 수량 − (승인 시점) 재고 수량
  - 실 생산량 = `ceil(부족분 / 수율)`
  - 총 생산 시간 = 평균 생산시간 × 실 생산량
  - 불량품은 없다고 가정한다 (수율은 계획 생산량 산정에만 사용, 실제 생산 결과는 전량
    정상품으로 처리).
- 생산 시간은 **현실 시간**을 기준으로 진행되며, 프로그램이 종료되어도 생산은 계속
  진행된 것으로 간주한다 (재실행 시 경과 시간을 반영하여 완료 여부를 판단해야 한다).
- 생산 완료 시 해당 주문 상태를 `PRODUCING` → `CONFIRMED`로 변경한다.
- 생산 현황 화면에는 현재 생산 중인 주문 정보와 대기 큐(FIFO 순서) 목록을 표시한다.
  세부 표시 항목(진행률 등)은 자유롭게 결정한다.

### 7.7 출고 처리

- `CONFIRMED` 상태 주문 목록에서 특정 주문을 선택하여 출고를 실행한다.
- 처리 후 주문 상태를 `RELEASED`로 전환한다.

## 8. 비기능 요구사항

- **데이터 영속성**: 애플리케이션을 재실행해도 시료/주문/생산 큐 데이터가 유지되어야 한다.
  저장 방식은 DataPersistence PoC에서 정의한 JSON 파일 기반 Repository 패턴을 따른다.
- **UI**: 콘솔 화면 구성은 자유롭게 결정하되, PDF의 예시 화면을 참고한다.
- **아키텍처**: MVC(Model/Controller/View) 구조를 따르며, 세부 패키지 구조는 MVC
  skeleton PoC를 참고한다.
- **개발 프로세스**: TDD(Red-Green-Refactor) 필수, 문서(PRD.md/CLAUDE.md) 최신화,
  의미 있는 단위의 커밋 이력 관리.

## 9. 기술 스택

- 언어: C++
- IDE/빌드: Visual Studio (vcxproj, MSBuild)
- 테스트 프레임워크: GoogleTest (GTest)
- 데이터 저장: JSON 파일 (커스텀 JSON 파서, DataPersistence PoC 참고)

## 10. 가정 및 제외 범위

- 동시성(멀티 유저 동시 접속)은 고려하지 않는다 — 단일 콘솔 세션 기준.
- 생산 라인은 항상 1개(단일 라인)이다.
- 인증/권한 관리는 범위 밖이다.
- 네트워크/서버 통신은 없으며, 로컬 프로세스 + 로컬 파일로 완결된다.
